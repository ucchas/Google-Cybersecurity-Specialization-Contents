Cybersecurity Incident Report Identification

Section 1: Identify the type of attack that may have caused this

network interruption

One potential explanation for the website's connection timeout error message is: SYN flood attack

The logs show that: Large number of TCP SYN requests coming from an unfamiliar IP address

This event could be: a SYN flood DoS attack targeting the server

Section 2: Explain how the attack is causing the website to malfunction

When website visitors try to establish a connection with the web server, a three-way handshake occurs using the TCP protocol. Explain the three steps of the handshake:

1.  A [SYN] request is sent from the source/client

2. A [SYN, ACK] packet is sent as a response from the web server to client

3. An [ACK] packet is sent back from the client side to complete the TCP connection

Explain what happens when a malicious actor sends a large number of SYN packets all at once: large number of SYN packets from fake IP addresses overwhelm the server and the server shuts down or becomes inoperable

Explain what the logs indicate and how that affects the server: The logs indicate that it is a DoS attack by intiating a SYN flood attack from a fake IP address. The large number of requests sent from the destination servers overwhelm the target server and the target server becomes inoperable

Security incident report

Section 1: Identify the network protocol involved in the incident

From the observed tcpdump log data, it can be seen that the first malicious incident which was the request sent from IP your.machine.36086 to initiate the download of the malware from the original website yummyrecipesforme.com with IP 203.0.113.22. The second malicious incident of redirecting to another website greatrecipesfrome.com with IP 192.0.2.172 was initiated from IP your.machine.56378. Both of these incidents occured on the port 80 which is the commonly used port for HTTP. Hence, the network protocol involved in the incident indeed the HTTP protocol

Section 2: Document the incident

The first malicious incident started after the successful DNS resolution request was resolved and the original IP 203.0.113.22 was detected in response. As soon as the DNS resolution request was resolved, the download of the malware from the original website IP 203.0.113.22 initiates on a different port on the source computer(port 36086), whereas, the request for DNS resolution was sent from port 52444. A lot of traffic is observed on the HTTP network protocol.

After about two minutes, the second incident occurs when the browser’s originally requested port number is seen again in the logs. The logs show the redirected address greatrecipesforme.com for which the DNS resolution response was IP 192.0.2.17. This is the where the spoofed website is hosted.

After the completion of redirection to the spoofed website, we can see about 5 minutes of delay from 14:20:32.204389 to 14:25:29.576492 when no packet was sent or received. Then, we can again see the packets being sent or acknowledged from the spoofed website, all of which was transferred using the HTTP network protocol.

Section 3: Recommend one remediation for brute force attacks

Various OS hardening measurements can secure the browser on the user’s end to remediate these kinds of brute force attacks. Application hardening and network hardening tactics such as threat detection and alarms, secure passport policies etc. needs to be implemented in order to ensure that these kinds of attacks never happen again.
